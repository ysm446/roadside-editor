# UV空間での侵食シミュレーション

道路・路肩のようなリボン形状に対して、ワールド空間ヘイトフィールドではなく **UV空間上で侵食計算を行う** 手法の技術メモ。

---

## 1. 概要と動機

### 1.1 やりたいこと

センターラインからスイープした道路メッシュと、その両脇の路肩リボンをUV展開する。UV空間を2Dグリッドとみなしてハイトマップを持たせ、その上で hydraulic erosion を回す。結果を displacement / mask テクスチャとして書き出す。

### 1.2 ワールド空間HFに対する優位

| 項目 | UV空間 | ワールド空間HF |
|---|---|---|
| 解像度配分 | コリドー上に一定の高密度 | 全域一様（無駄が大きい） |
| ヘアピン・立体交差 | XY上で重なる領域を扱える | 原理的に不可能 |
| 出力形式 | そのまま displacement / mask | 転写処理が必要 |
| 周囲地形との連続性 | 境界フラックス実装が必要 | 自然に連続 |
| 垂直に近い法面 | 破綻する | 破綻する |

**計算量の比較例**（10km の道路、路肩幅 2.5m、4cm/texel）

- UV空間: 250,000 × 64 ≈ **1,600万セル**
- ワールドHF（2km四方を同解像度）: ≈ **25億セル**

2桁以上の差。加えて、UV空間では水の流路長が「路肩を横切る数m」で完結するため、地形全体で流域を形成させるフルシミュレーションに比べ必要反復数も桁で少ない。

### 1.3 適用範囲の前提

- 法面が **垂直に近づかないこと**（$N_z \to 0$ で破綻）
- 擁壁・切土の急崖は本手法の対象外
- 面はワールドZに対しておおむね高さ場であること

## 2. UV歪みの解析

### 2.1 リボンのヤコビアン

一般的なUV展開と異なり、リボン形状ではヤコビアンが閉じた形で書ける。

センターライン $C(s)$、横方向オフセット $v$、曲率 $\kappa(s)$、横方向単位ベクトル $B(s)$ として

$$P(u,v) = C(s) + v\,B(s)$$

$$\left|\frac{\partial P}{\partial u}\right| = \bigl(1 - \kappa(s)\,v\bigr)\frac{ds}{du}$$

- **内カーブ側**（$v < 0$ 方向）: 圧縮
- **外カーブ側**（$v > 0$ 方向）: 伸長
- $v \ge 1/\kappa$ でオフセット曲線がカスプを生じ自己交差

$U$ を弧長パラメータにしておけばセンターライン上は歪みゼロ。路肩幅数m に対して曲率半径が数十m以上あれば、誤差は数%〜十数%程度に収まる。

### 2.2 メトリック（第一基本形式）

$$E = |P_u|^2,\quad F = P_u \cdot P_v,\quad G = |P_v|^2$$

**リボンでは $F = 0$**（接線 $T$ と横断方向 $B$ が直交）。よってメトリックは対角となり、逆メトリックの計算が不要。スケール係数2つで済む。

さらに分離可能:

$$s_u(u,v) = \bigl(1 - \kappa(u)\,v\bigr)\frac{ds}{du}, \qquad s_v = \text{const}$$

$\kappa$ は $u$ のみの関数、$v$ は列インデックスそのもの。**1Dの曲率配列1本**で全テクセルのスケールが復元でき、2Dチャンネルとして保持する必要すらない。

## 3. ソルバへの補正項

### 3.1 ベイクすべき量

| 量 | 内容 | 用途 |
|---|---|---|
| `P(u,v)` | ワールド座標 | 位置参照 |
| `P_z` | ワールドZ | 重力ポテンシャル |
| `N_z` | 法線のZ成分 | 重力ポテンシャル |
| `E, F, G` | 第一基本形式 | 勾配・面積 |
| `dA` | $\sqrt{EG - F^2}\,du\,dv$ | 水量・堆積量の重み |

### 3.2 勾配計算

$$\nabla h = g^{-1}\begin{bmatrix} \partial_u h \\ \partial_v h \end{bmatrix}$$

リボンの場合は $F = 0$ なので単純に

$$\frac{\partial h}{\partial x_{\text{world}}} = \frac{1}{s_u}\frac{\partial h}{\partial u}, \qquad \frac{\partial h}{\partial y_{\text{world}}} = \frac{1}{s_v}\frac{\partial h}{\partial v}$$

### 3.3 流量計算

D8 でも pipe model でも、「隣接セルへの距離」を $du$ ではなく実距離 $s_u\,du$ に置き換える。これだけでも見た目のオーダーはかなり合う。

---

## 4. 重力の扱い（最重要）

侵食を駆動するポテンシャルは、法線方向の変位 $h$ **ではなく、ワールドZ座標**である。

道路にはバンク（片勾配）と縦断勾配があるため、UVマップに変位だけを入れて侵食をかけると「水が道路の傾きを無視して流れる」ことになる。

ソルバに渡す「標高」は以下を使う:

$$\phi(u,v) = P_z(u,v) + h(u,v)\,N_z(u,v)$$

これにより

- 片勾配で水が路肩へ集まる
- 縦断勾配に沿って流れる
- 法面を落ちていく

という挙動が自然に発生する。

**制約**: $N_z \to 0$（法面が垂直に近い）で破綻する。§1.3 の前提と同じ。

---

## 5. パフォーマンス

### 5.1 ALUコストはほぼ無視できる

1. **メトリックが静的** — 形状不変なら反復ループの外で一度焼くだけ。侵食ソルバは典型的にメモリ帯域律速のため、テクセルあたり数回の乗算増は実測でほぼ現れない。
2. **メトリックが対角** — 逆行列不要。
3. **1D配列で復元可能** — L1に載るサイズ。実質コストゼロ。

追加で常時読むのは `P_z` と `N_z` の2チャンネル。これがコスト増の主因だが、読み取り専用かつアクセスパターンが完全にコヒーレント。**体感で1〜2割程度**。

### 5.2 本当の落とし穴: CFL条件

陽解法では安定条件が **最小セルサイズ** に支配される。

ヘアピン内側で $1 - \kappa v$ が 0.3 まで縮むと、**そのテクセル1個のためにグリッド全体の $dt$ が 1/3 になる**（反復回数3倍）。

**対策**

- **$s_u$ に下限クランプ**（推奨） — 内カーブの侵食が過小になるが、そこは元々土砂が溜まる側なので影響は小さい
- $u$ を内側エッジ基準の等弧長で再パラメータ化し、最小セルを揃える
- ローカルタイムステップ（セルごとのサブサイクル）

多くの場合クランプで十分。

## 6. UVレイアウト依存の歪み

### 6.1 補正できるもの / できないもの

| 歪みの種類 | 補正 | 実際どうなるか |
|---|---|---|
| 全体スケール / シェル別スケール | ○ 完全 | 係数一発。$s_u, s_v$ に吸収される |
| テクセル密度の差 | △ 物理のみ | 物理は正しいがディテールが揃わない |
| アスペクト比（非正方テクセル） | ○ だが劣化 | 数値拡散が方向に偏る |
| シェル境界（シーム） | 要実装 | 放置すると水と土砂が消える |

$s_u, s_v$ をベイクした時点で **UVの単位系はソルバから見えなくなる**。UVは単なるインデックス空間で、物理は全部メトリック側に入る。0〜1正規化でもシェルごとに倍率が違っても問題ない。

### 6.2 密度差は「正しいけど揃わない」

あるシェルを半分の密度でパックすると、メトリック補正により **物理量としては正しい結果** が出る（流量・勾配・傾斜角すべて合う）。

しかし侵食は不安定性の増幅であり、**リル（雨溝）の間隔は解像度に支配される**。最小解像波長がそのままガリーのピッチになる。

結果として、隣り合うシェルで「物理的には整合しているのに、溝の細かさが目に見えて違う」という絵になる。テクスチャで見ると密度差がバレる。

**これは係数では直せない。** 密度を揃えるか、密度に応じてノイズシード／侵食強度をスケールするかのどちらか。

### 6.3 CFLは最悪値に引きずられる

アトラス全体を1ソルバで回すと、一番細かいテクセルが全体の $dt$ を決める。細かくパックされたシェルが1枚あるだけで、他の全シェルが無駄に細かい時間刻みで回る。

**対策**: シェルごとに独立して解く。侵食は局所的なので、シェル間で交換するのは境界フラックスのみ。それぞれが自分の $dt$ とステップ数を持てる。

### 6.4 推奨: ワールド比例UVを強制する

UVレイアウトの段階で「0〜1にフィット」ではなく **「1テクセル = 4cm」を固定してパック** する。Houdini なら UV Layout の Scale を Uniform にし、シェルごとの自動スケーリングを切る。

得られるメリット:

- $s_u, s_v$ がカーブの $1-\kappa v$ 項以外ほぼ1になり、補正がほぼ不要
- CFLが均一
- リルのピッチが全域で揃う
- テクスチャ解像度の割り当てが直感的

アトラス面積は多少無駄になるが、地形テクスチャなら問題にならない。

## 7. 境界の扱い

### 7.1 リボン間の接続

道路・路肩・法面を別リボン＝別UVタイルにすると、水とシルトがタイル境界で消える。

- **推奨**: 道路〜路肩〜法面裾までを **一枚の連続UVアトラス** にする（$v$ 方向に繋げるだけなので容易）
- 代替: 各タイルにゴーストセルを持たせ、毎ステップ流量を交換

### 7.2 周囲地形への吐き出し

法面の裾から先の既存ヘイトフィールドへの土砂供給は、境界での堆積フラックスを別途ワールド空間HFに転写する処理が必要。

---

## 8. 実装方針

### 8.1 簡易版

歪みが小さい前提（緩いカーブ、幅の狭い路肩）なら、**UVを2Dヘイトフィールドにマップして HF Erode をそのままかける**。

$1 - \kappa v$ が **0.85〜1.15 の範囲** なら、雨溝の見た目としては視認できない。

### 8.2 完全版

ヘアピン・幅の広い法面を含む場合は、OpenCL / VEX で自前ソルバを書きメトリック補正を入れる。

### 8.3 検証手順

侵食を回す前に必ず通すチェック:

1. $\sqrt{EG - F^2}$ を色でベイクして可視化 — 均一なら真っ平ら、問題があれば一目でわかる
2. **UV上の総面積 × スケール補正** と **実メッシュの総面積** を比較 — メトリックのベイクミスを検出できる

---

## 9. 汎用化の余地

`P(u,v)` と `N(u,v)` をベイクしてしまえば、リボン以外にも同じ枠組みが使える。

- 岩壁
- 擁壁
- 階段状の造成地

汎用の **「サーフェス上侵食」ノード** として切り出せる形。ただし §4 の $N_z \to 0$ 制約は共通して残る。
